<a id="프로젝트-개요"></a>
<details open>
<summary><b>📌 프로젝트 개요</b></summary>
<br>
  
 ### 프로젝트 배경
  국내 구독 시장의 성장으로 사용자 1인당 평균 3개 이상의 서비스를 이용하는 것이 일반화되었지만 
  서비스가 플랫폼별로 분산되어 있어 구독 정보와 결제 내역을 한눈에 파악하기 어려운 문제가 발생하고 있습니다. 
  이로 인해 무료 체험 종료 후 자동 결제, 사용하지 않는 구독의 지속 결제, 유사 서비스의 중복 구독 등
  다양한 비효율이 발생하고 있으며, 사용자 피로도 또한 증가하고 있습니다.
  
  이러한 문제를 해결하기 위해, 여러 구독 정보를 한곳에서 통합 관리하고 지출을 효율적으로 관리할 수 있는 올인원 구독 관리 서비스의 필요성이 증가하고 있습니다.<br><br>
  
  ### 주요 기능
  
  💳 구독 자산 관리<br>
  구독 서비스의 결제일, 금액, 결제 주기 등 핵심 정보를 통합 관리하고, 구독 상태(활성 / 해지)를 기준으로 전체 구독 현황을 한눈에 조회할 수 있도록 설계했습니다.
  
  📊 소비 분석 및 알림<br>
  월별 및 카테고리별 구독 지출을 시각화하고, 소비 패턴 분석과 결제 예정일 알림 기능을 통해 체계적인 지출 관리를 지원하도록 구성했습니다.
  
  💬 구독 커뮤니티<br>
  사용자 간 구독 경험과 정보를 공유할 수 있도록 커뮤니티 기능을 제공하며, 게시글 저장 및 외부 공유가 가능한 구조로 설계했습니다.
  
  🤖 맞춤형 추천 및 최적화<br>
  사용자의 소비 패턴과 구독 데이터를 기반으로 유사 서비스 분석 및 최적의 구독 조합을 제안할 수 있도록 설계했습니다.
  </details>

<a id="서비스-목표"></a>
<details>
<summary><b>🎯 서비스 목표</b></summary>
<br>

- 사용자가 여러 구독 서비스를 한 화면에서 쉽게 조회하고 관리할 수 있는 UI를 제공합니다.
- 결제 예정일, 카테고리별 소비, 월별 추이 등을 직관적으로 확인할 수 있도록 시각화합니다.
- 인증 상태, 입력 검증, 예외 처리까지 포함한 안정적인 사용자 흐름을 제공합니다.
- 백엔드 기능을 실제 서비스 경험으로 연결하는 프론트엔드 구조를 구축합니다.

</details>

<a id="주요-기능"></a>
<details>
<summary><b>🧩 주요 기능</b></summary>
<br>

### 1. 인증 및 세션 관리
- 로그인 / 회원가입 화면 제공
- JWT 기반 로그인 세션 저장
- 토큰 만료 시 세션 정리 후 로그인 화면으로 이동
- 비로그인 사용자의 접근 제한 라우팅 처리

### 2. 대시보드
- 월별 소비 요약
- 카테고리별 소비 분석
- 결제 예정 구독 목록
- 인사이트 패널 및 홈 프리뷰 연계

### 3. 구독 관리
- 구독 목록 조회
- 구독 상세 패널 표시
- 카테고리 / 결제 주기 / 금액 입력 기반 구독 추가
- 서비스 이미지 자동 매핑

### 4. 결제 캘린더 및 알림
- 달력 기반 결제 일정 확인
- 선택 날짜 상세 내역 패널
- 소비 추이 패널
- 알림 센터 및 헤더 알림 팝오버 제공

### 5. 커뮤니티 및 AI 추천
- 게시글 CRUD 및 스크랩 기능
- 추천 조건 입력, 결과 조회, 저장, 기록/상세 조회
- FAQ 페이지 제공

</details>

<a id="실행-방법"></a>
<details>
<summary><b>🚀 실행 방법</b></summary>
<br>

### 1. 실행 환경
- Node.js 18 이상
- npm
- 백엔드 서버 실행 상태 필요

### 2. 설치
```bash
npm install
```

### 3. 환경 변수 설정
프로젝트 루트에 `.env` 파일을 생성합니다.

```env
VITE_API_BASE_URL=http://localhost:8080
```

### 4. 개발 서버 실행
```bash
npm run dev
```

### 5. 배포용 빌드
```bash
npm run build
```

### 6. 미리보기 실행
```bash
npm run preview
```

</details>

<details>
<summary><b>📝 프로젝트 회고 및 향후 확장</b></summary>
<br>


### 팀원 회고
- **김가영**:
  >이번 프로젝트를 통해 처음으로 프론트엔드에서 사용자 화면을 직접 구성하고, 백엔드 API와 연결하는 과정을 경험했다. 단순히 화면을 만드는 것을 넘어서, 사용자의 입력이 어떤 식으로 처리되고 서버와 데이터를 주고받는지 전체 흐름을 이해할 수 있었다. 처음 구성하는 화면에 막막함을 크게 느끼고 figma를 통한 화면설계서 제작과정에 정말 많은 시간을 쏟아부으며 기본적인 틀을 잡아가며 익숙해지려 노력했다.
특히 상태 관리와 API 연동을 직접 다뤄보면서, 화면에 보이는 요소 뒤에서 어떤 로직이 동작하는지에 대해 많이 고민하게 되었다. 작은 기능 하나라도 자연스럽게 동작하도록 만들기 위해 조건 처리나 예외 상황을 신경 써야 했고, 이를 통해 사용자 경험을 고려한 개발의 중요성을 느낄 수 있었다.
처음에는 화면 구현에만 집중했지만, 점점 구조를 생각하면서 코드를 작성하게 되었고, 그 과정에서 컴포넌트 분리나 코드 재사용성의 필요성도 체감할 수 있었다. 이번 경험을 통해 프론트엔드는 단순히 디자인을 구현하는 영역이 아니라, 서비스 흐름을 만들어가는 중요한 역할이라는 점을 이해하게 되었다.
  <br>
  
- **김다솜**:
  >프론트 작업을 하면서 구조를 먼저 설계하는 것이 중요하다는 점이었습니다. 이번 프로젝트에서는 캘린더, 소비 분석, 카드 관리처럼 한 화면 안에서도 여러 기능과 API가 함께 연결되어 있었기 때문에, 부모 컴포넌트와 자식 컴포넌트의 역할이 명확하지 않으면 상태 관리와 데이터 흐름이 꼬이곤 하였습니다. 실제로 구현 과정에서도 어떤 데이터는 부모에서 관리하고 어떤 동작은 자식에서 처리할지 정리가 되지 않아 수정 범위가 계속 커졌고, 백엔드와 연동할 때도 응답 구조와 화면 구조가 맞지 않아 어려움을 겪었습니다. 이를 통해 프론트엔드 개발에서도 단순히 UI를 나누는 것이 아니라, 컴포넌트의 책임과 상태의 흐름을 먼저 설계하는 과정이 중요하다는 점을 배울 수 있었습다.
이번 프로젝트를 통해 프론트엔드에서도 화면 구현보다 컴포넌트 책임과 상태 흐름을 먼저 설계하는 과정이 중요하다는 점을 배웠습니다.

또 API 명세서를 보고 데이터 구조를 파악하는 능력이나, 상태 관리를 통해 화면에 반영하는 방식에 따라 구현 결과가 달라진다는 것도 알게 되었다. 다음에는 이 부분을 더 신경써서 구현해야겠다.
